#01
GitLab — Sanitization Bypass via data- Attributes (XSS+CSRF)
HIGH STORED XSS+CSRF HackerOne · $3,000+
SCENARIO
GitLab uses DOMPurify (v-safe-html) but forgets to sanitize data-disable-with attributes used by Rails-ujs. Inject an invisible clickable layer over the whole page to trigger CSRF via XSS.
// EXECUTION CHECKLIST
  • Identify all input fields that render user HTML (Wiki, CI/CD Job names, Issue descriptions)
  • Test basic XSS: <script>alert(1)</script> — confirm it's filtered by DOMPurify
  • Check which data-* attributes survive: try data-remote, data-url, data-disable-with
  • Inject invisible overlay payload: <a class="fixed-top fixed-bottom" data-disable-with="<img src=x onerror=fetch('/api/...')>">
  • If alert() is blocked by CSP, chain to <form> CSRF to change Admin password instead
  • Verify: rails-ujs triggers data-disable-with on any click — escalate to Admin CSRF

"DOMPurify blocks tags and events, but it cannot know which data attributes a 3rd-party library will execute. The sanitizer's allow-list is your enemy's whitelist."

RECON
Recon — Map All State-Changing Endpoints
RECON METHODOLOGY Pre-Attack SOP
SCENARIO
Before any CSRF test, you must know every endpoint that changes server state. Most CSRF reports fail because hunters test the wrong endpoints. This is your mandatory first step.
// EXECUTION CHECKLIST
  • Open Burp Suite → set scope to target → browse all features while logged in
  • In Proxy History: filter by POST, PUT, PATCH, DELETE methods only
  • List all sensitive actions: email change, password change, delete account, add SSH key, link social account
  • For each: check if CSRF token is in body/header, or if only cookies are used for auth
  • Check SameSite attribute on session cookies: Strict=safe, Lax=partially safe, None=CSRF possible
  • Note any /api/mobile/ or /api/v2/ endpoints — these often lack CSRF protection added to web flows

"The best CSRF hunters don't find one bug. They map the whole attack surface first, then systematically test every state-changing action. Speed comes after methodology."

PHASE 03
TOKEN BYPASS — Break the CSRF Protection Itself
Bypass CSRF tokens through validation logic flaws and token weaknesses
#07
Shopify — Empty Token Validation Bypass
HIGH EMPTY TOKEN HackerOne #114115
SCENARIO
Backend code checks if (params.has("token")) validate() — not the token's value. Empty string passes. Send authenticity_token= (blank) and the server processes the request.
// EXECUTION CHECKLIST
  • Intercept request with CSRF token → note token name (e.g., authenticity_token)
  • Test 1: remove the token entirely → expect 403 Forbidden
  • Test 2: send wrong token value → expect 403 Forbidden
  • Test 3 (KEY): send token with empty value: authenticity_token= → observe response
  • Also fuzz: token=null, token=true, token=0, token=[] (array bypass in PHP)
  • If 200 OK with empty/null token → CSRF bypass confirmed, build full PoC

"isset() checks presence. empty() checks value. Developers who write `if token exists, validate` but not `if token is valid AND non-empty` leave a gaping hole."

#08
Fintech — CSRF via Full Parameter Removal (Soft Validation)
HIGH PARAM REMOVAL HackerOne Private · $2,800
SCENARIO
Server uses conditional validation: "IF csrf_token is present THEN validate it." If the entire parameter is absent, validation is skipped entirely. Delete the token parameter = bypass.
// EXECUTION CHECKLIST
  • Normal flow → 200 OK. Wrong token → 403. Now: delete the entire token parameter from request body
  • If response is 200 OK → "Soft Validation" confirmed — server only validates when token is present
  • If token is in a HEADER (e.g., X-CSRF-Token): delete the entire header row, not just its value
  • Check mobile API endpoints separately: /api/mobile/ often forgets token enforcement
  • Build PoC HTML form WITHOUT any token field — confirm victim action executes

"Optional validation is the enemy of security. 'If token exists, check it' means 'no token = no check'. Always verify that token absence returns HTTP 403, not 200."

#09
Social Network — Attacker's Token Accepted (No Session Binding)
HIGH TOKEN BINDING HackerOne Public · $2,000
SCENARIO
Server validates that token was generated by the system, but NOT that it belongs to the current user's session. Attacker's own valid token works for any user. "Global Token Pool" bug.
// EXECUTION CHECKLIST
  • Create 2 accounts: Attacker (Account A) and Victim (Account B)
  • From Account A: capture a request with a valid csrf_token value → copy it
  • From Account B (Victim): intercept sensitive request → replace its token with Account A's token
  • If 200 OK → Token-Session binding missing — server only checks validity, not ownership
  • Check if token is reusable (one-time use? If same token works twice → another bug)
  • Build PoC: hardcode attacker's valid token in the CSRF form — it works on any victim

"The token must be bound to the session. A valid token for User A must be invalid for User B. This seems obvious, but global token pools are found regularly in large codebases."

#04
Fintech — JSON CSRF via text/plain Content-Type Confusion
HIGH CONTENT-TYPE HackerOne Private · $2,500
SCENARIO
API uses JSON. Developers think JSON = CSRF-safe (HTML forms can't send JSON). But server accepts text/plain and parses the body as JSON anyway. HTML form + enctype="text/plain" = no CORS preflight = CSRF!
// EXECUTION CHECKLIST
  • Intercept JSON API request → in Burp Repeater: change Content-Type: application/json to text/plain
  • If server returns 200 OK → it's not enforcing Content-Type strictly
  • Build HTML form PoC with enctype="text/plain" and craft the JSON body using input name trick:
  • Form input: name='{"email":"h@evil.com","ignore":"' value='1"}' → browser sends valid JSON
  • Key: text/plain is a "simple" content-type → browser sends NO CORS Preflight OPTIONS request
  • Confirm victim's data changes when they visit attacker's page while logged in

"text/plain is the CSRF hunter's best friend in 2026. It triggers no CORS preflight, can carry JSON body, and many backend parsers don't care about the Content-Type header."

#12
SaaS Platform — JSON CSRF via Content-Type Laxity
HIGH JSON CSRF HackerOne Private · $3,000
SCENARIO
Server's JSON body-parser auto-detects JSON from body content (starts with {) regardless of Content-Type header. No preflight means cross-origin form can submit JSON without CORS blocking.
// EXECUTION CHECKLIST
  • Delete the Content-Type header entirely from JSON request → does server still process it?
  • Try: Content-Type: application/x-www-form-urlencoded with JSON body → does it parse?
  • If yes: build HTML form with enctype="text/plain". Use ignore field to absorb the = sign
  • Confirm no CORS Preflight: in browser devtools Network tab, OPTIONS request should be ABSENT
  • Verify impact: check if address, email, or API key changes in victim's account

"Body-parser 'content sniffing' — detecting JSON from body instead of header — is the root cause. Any framework that looks at { before checking Content-Type is vulnerable."

#15
Cloud Provider — CSRF via No Content-Type Enforcement
HIGH MISSING ENFORCE HackerOne Public · $3,500
SCENARIO
API endpoint requires JSON. Server does NOT return 415 Unsupported Media Type when Content-Type is missing or wrong. It silently parses the body and executes the action — full CSRF via simple form.
